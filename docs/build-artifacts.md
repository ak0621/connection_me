# Build Artifacts

This project uses one C++20/CMake/Qt codebase for macOS, Windows, and Linux. The executable protocol and config format are shared, so binaries built from the same commit can interconnect across the LAN.

## LocalSend-Style Packaging Logic

The packaging flow mirrors the useful parts of LocalSend's release model while keeping this project on C++/CMake:

- build native artifacts on matching OS runners instead of cross-building every installer from one host;
- keep the desktop UI as the primary runnable app when Qt is available;
- publish platform-explicit package names such as `MyBarrier-0.2.0-linux-x86-64.deb`;
- provide both installer formats and portable archives;
- build Linux AppImage from an `AppDir` plus `packaging/appimage/AppImageBuilder_x86_64.yml`.

## Platform Build Commands

Linux/macOS:

```bash
scripts/build.sh
scripts/package-current.sh
```

Windows PowerShell:

```powershell
.\scripts\build.ps1
.\scripts\package-current.ps1
```

Linux AppImage on a Qt-enabled Linux host:

```bash
scripts/package-appimage.sh
```

Expected executable outputs:

| Platform | CLI executable | Desktop UI when Qt is available |
|----------|----------------|-------------------------------|
| Linux | `build/mybarrier` | `build/MyBarrier` |
| macOS | `build/mybarrier` | `build/MyBarrier.app` |
| Windows Visual Studio generator | `build/Release/mybarrier.exe` | `build/Release/MyBarrier.exe` |
| Windows single-config generator, such as Ninja | `build/mybarrier.exe` | `build/MyBarrier.exe` |

Expected package and installer outputs:

| Platform | Runnable package | Installer/package formats |
|----------|------------------|---------------------------|
| Linux | `dist/MyBarrier-<version>-linux-x86-64/`, `.tar.gz` | `build/MyBarrier-<version>-linux-x86-64.deb`, `.tar.gz`, `.zip`, CI AppImage |
| macOS with Qt UI | `dist/MyBarrier-<version>-macos-arm-64/` or `dist/MyBarrier-<version>-macos-x86-64/` | `build/MyBarrier-<version>-macos-*.dmg`, `.tar.gz`, `.zip` |
| macOS CLI-only | CPack `.tar.gz` and `.zip` | no DMG |
| Windows | `dist/MyBarrier-<version>-windows-x86-64/`, `.zip` | `build/MyBarrier-<version>-windows-x86-64.exe`, `.zip` |

Installer packages are generated by CPack:

```bash
cmake --build build --target package --config Release
```

The desktop UI is controlled by `MYBARRIER_BUILD_GUI`:

- `AUTO`: build the Qt UI if Qt 6 Widgets is found, otherwise build CLI only.
- `ON`: require Qt 6 Widgets and fail configuration if it is missing.
- `OFF`: build CLI only.

For macOS, the DragNDrop DMG generator is enabled only when the Qt UI target exists. That keeps CLI-only builds from producing a misleading DMG with just `bin` and `share` folders.

On Windows, the NSIS installer generator requires NSIS to be installed. The GitHub Actions workflow installs it before packaging.

## Runtime Log Window

The Qt desktop UI opens a default `Run Log` dock on every supported desktop platform. It records:

- GUI startup platform and config home;
- resolved CLI executable path;
- daemon command line, stdout/stderr, PID, stop events, and exit code;
- every UI-triggered CLI command with stdout, stderr, and exit code.

This is the runtime visibility layer for the current interactive app while native input capture/injection is still being implemented.

## CI Artifacts

The workflow `.github/workflows/build.yml` builds and uploads:

- `MyBarrier-linux-x86-64`
- `MyBarrier-macos-arm-64`
- `MyBarrier-windows-x86-64`

Each uploaded artifact contains a runnable `dist/MyBarrier-<version>-<platform>-<arch>/` directory plus CPack packages from `build/MyBarrier-*`. The workflow installs Qt and configures `-DMYBARRIER_BUILD_GUI=ON`, so the runnable package and installers include the desktop UI: Linux `MyBarrier`, macOS `MyBarrier.app`, and Windows `MyBarrier.exe`. Linux CI also uploads `*.AppImage`.

The macOS app bundle produced by public CI is ad-hoc signed during packaging, but it is not notarized with an Apple Developer ID. On first launch, macOS Gatekeeper can still require a manual trust override. For a package downloaded from this repository, remove the quarantine flag after dragging the app to Applications:

```sh
sudo xattr -dr com.apple.quarantine /Applications/MyBarrier.app
open /Applications/MyBarrier.app
```

A Linux machine cannot directly produce a native macOS executable without an Apple toolchain. Windows binaries can be cross-compiled in some setups, but this project uses Windows CI or a Windows machine as the reliable path.

## Runtime Interop

To interconnect devices, run the same-version `mybarrier` binary on each OS:

1. Start one daemon with `serve --pair-code <code>`.
2. Pair from another device with `pair --host <ip> --code <code>`.
3. Use `ping --peer <peer>` to verify the trusted peer connection.
4. Use clipboard and file commands across paired peers.

Example:

```bash
./mybarrier serve --name mac --role server --pair-code 123456
./mybarrier pair --host <mac-ip> --code 123456
./mybarrier ping --peer mac
```

Native keyboard/mouse sharing still requires platform input adapters. The current binary has the authorized input event channel and diagnostics, but native injection is not complete.
